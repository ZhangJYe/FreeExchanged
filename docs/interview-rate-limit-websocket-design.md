# 面试指南：限流设计与 WebSocket 架构体系

本指南专注于解析当前网关层的高可用利器（限流）以及实时数据推送核心（WebSocket）的设计原理与落地细节。并围绕这两大技术点梳理了多道具有高区分度的面试题。

---

## 一、高可用防线：限流与降级设计

在我们的微服务架构中，Gateway（API 网关）作为抵御洪峰流量的第一道防线，承担了极其关键的限流降级任务。我们并没有自己造轮子，而是深度融合了 go-zero 框架原生提供的高效双重限流机制。

### 1.1 双重限流机制的落地

#### 第一层：自适应限流 (Adaptive Shedding)
*   **配置方式**：在 `gateway.yaml` 等配置文件中设置 `CpuThreshold: 900`（代表 90% CPU 占用率阈值）。
*   **设计原理**：这是一种非常聪明的限流方式。相较于传统的 QPS 手动预估限流，自适应限流不需要人工拍脑门设定阈值。go-zero 会实时在后台协程监控宿主机的 CPU 使用率。当 CPU 使用率越过设定的安全水位线（如 90%）时，系统会自动启用丢弃请求策略（Shedding），利用特定算法阻断部分请求，直接返回 `429 Too Many Requests`。
*   **核心优势**：在流量洪峰或者遭遇突发未知压力时，完全保护 CPU 不被持续的高负载打挂（雪崩效应），实现服务的极限稳定。

#### 第二层：并发限流 (MaxConns)
*   **配置方式**：在 `gateway.yaml` 设置 `MaxConns: 10000`（最大并发连接数）。
*   **设计原理**：Gateway 控制同时接客的最大并发度，如果突发的连接数超爆了预设缓冲池，对超出的请求直接斩断，返回 `503 Service Unavailable`。
*   **核心优势**：限制 Gateway 层向更下游业务 RPC（譬如 Ranking 排行榜、DB）施加并发压力。将长队拦在系统的边缘层，以极小的代码开销换取后备隐藏服务的绝对安全。实测压测数据表明，在被限流情况下，Gateway 层的 P99 延迟可能飙升到上百毫秒（排队抢锁），而底层的 RPC 却毫无波澜地稳在 4ms 内。

### 1.2 可观测闭环
单单有限流是不够的，如果看不见拦截了多少就是薛定谔的限流。得益于 go-zero 内置的 Prometheus 支持，每一次限流拦截都会体现为 Grafana 大盘上的数字跳动，核心埋点监控如：
*   `http_server_requests_drop_total`：记录被丢弃请求
*   `rate(http_server_requests_code_total{code=~"429|503"}[1m])`：直观展示拦截速率与丢弃水位线。

## 二、底层推送核心：WebSocket 管理设计

为了实现行情 / 排行榜的实时数据下发，我们在网关层内置了独立的 WebSocket 服务器体系，剥离了短连接和长连接的处理。核心代码位于 `app/gateway/internal/ws` 目录下。

### 2.1 鉴权与协议升级 (`handler.go`)
1.  **突破浏览器限制的握手鉴权**：由于浏览器原生的 WebSocket API 并不支持像 XMLHttpRequest 那样自定义 HTTP Headers（塞入 `Authorization: Bearer <Token>` 失败），因此在我们的体系中做了一次降级抽象，**将 PASETO Token 通过 Query 参数传递** （`GET /ws?token=<paseto_token>`）。
2.  **纯粹算力鉴权**：在 HTTP 层通过 `TokenMaker.VerifyToken(tokenStr)` 解析令牌。整个鉴权不查表、没有 IO 延时损耗，极其轻量迅速。
3.  **Upgrade 升级协议**：鉴权通过后利用 Gorilla WebSocket 的 `Upgrader.Upgrade()`，从普通的 HTTP 一脚油门飙升为持久化的双向通信协议协程。

### 2.2 核心会话管理器 (`manager.go`)
1.  **线程安全的集中注册表**：通过一个全局挂载的 `Manager` 单例模式管理，内部采用一个核心字典 `clients map[int64]*websocket.Conn` 保存所有在线用户的连接映射。因为有并发读写，为其绑定了 `sync.RWMutex` 读写锁。
2.  **顶号逻辑（挤占旧连接）**：在 `Add(userId, conn)` 方法时，专门针对**单设备登录限制**设计了一段逻辑检查：若通过字典扫描发现这名 `userId` 之前有过旧连接，则调用 `old.Close()` 直接将其一脚踢下线（异地登录挤占），再关联新的 Conn。
3.  **死链接监控保活机制**：伴随着连接注册的成功，立刻启动一个专门陪伴的 Goroutine （`go readLoop`）。这个协程虽然大部分时间不管业务逻辑，但它阻塞监听对端发出的 Close 帧或断连错误。一旦检测到物理断开，协程生命结束，调用 `defer Hub.Remove(userId)` 优雅回收地图中的垃圾资源。

## 三、面试高频问题与对答演练 (Q&A)

### 🚀 限流篇

**Q1：你们项目里的并发处理是如何做的？如果瞬间有几倍于平常的流量涌进来，服务扛得住吗？**
> **答**：我们在网关入口处配置了双保险的防护机制来应对突增流量，防止了雪崩效应的发生。
> 首先是**并发限流 (MaxConns)**，我们明确配置了连接槽最大数，超出直接拒绝并返回 503，这种"简单粗暴"的方案像是在系统口岸建了一道闸门，确保底层负责计算的 RPC 服务（如排行的 Redis 拉取）免受无限制 IO 并发之苦。
> 再者我们使用了框架（go-zero）原生自带的最高级武器：**自适应动态限流 (Adaptive Shedding)**。不需要我通过人工拍脑门设定死板的 QPS 阈值，它会后台探查机器当前的 CPU 瓶颈，比如只要 CPU 水位跨过 90% 红线，立刻切入防守反击阶段按比例丢包（返回 429）。不仅如此，所有以上拦截产生的数据我们不仅知道，而且在 Promethus 和 Grafana 大盘上一览无遗。

**Q2：对于一些很容易被脚本刷量的接口（譬如“点赞”），如果我想针对它单独做精确的速率控制（比如每秒最多允许处理10个点赞），你们目前的架构怎么处理的？**
> **答**：目前系统已经专门针对这种情况**真实落地落地了令牌桶（Token Bucket）限流**。
> go-zero 框架原生自带的双重限流是全局保护的（自适应限流是被动保护CPU），无法做针对单个接口精确粒度的 QPS 管控。所以为了精确控制“点赞”等高危接口（如 `/v1/article/like`），我们在 Gateway 手写提供了一个自定义的 `RateLimitMiddleware` 中间件（底层使用了 `golang.org/x/time/rate` 实现的令牌桶算法）。
> 这样设计极其优雅：得益于由网关接管路由调度的责任链（Pipeline）机制，我们的限流逻辑完全与具体业务代码解耦。在这个真实项目中，我们在 `ServiceContext` 挂载了一个限制为 10 QPS / 容量为 100 的令牌桶中间件，只需要通过 api 文件的声明式注解 `@server (middleware: RateLimitMiddleware)` 即可精准保护点赞路由。一旦瞬间涌入的突发刷量请求使得桶中无令牌可用，网关层就会立刻熔断这批请求，返回 `429 Too Many Requests`。对于底层的 RPC 服务来说，它们感受不到任何超量负载，整个过程实现了真正的边界斩杀与对主业务零侵入的架构。

### 🚀 WebSocket 篇

**Q3：你介绍下长连接架构中是如何识别用户身份的？这个和正常的 API 接口做法一样吗？**
> **答**：核心鉴权算法一致，都是基于对称加密的 PASETO，但**传输方式做了降级变通**。普通的 API 请求是标准地携带在 `Header: Authorization` 里。但在接浏览器的 WebSocket 时，W3C 标准并不允许 JS 层的 WebSocket API 携带自定义 Header。所以我们选择将 Token 写进初次建立联络的 `Query` 字符串中，在还没 Upgrade 到 WebSocket 的 HTTP 拦截阶段，将其解析破译。一旦解密通过拿到了用户 ID (uid)，再批准底层协议升级和映射注册，全过程不会增加任何 DB IO。

**Q4：我看到你们项目定义了一个全局映射字典 (`clients map[int64]*websocket.Conn`) ，你觉得这种设计的潜在隐患是什么？当用户量非常多的时候怎么优化呢？**
> **核心痛点（面试鉴别度极高，属于必杀技）**：
> **答**：目前的实现在**单节点网关**状态下是极其高效的，用 Map + 读写锁 (`sync.RWMutex`) 支撑单机几万量级在线没有问题。但这存在一个致命的扩展性隐患：**无法水平扩展构建分布式横向扩容网关**。
>
> **为什么有隐患？**
> 如果我们的网关扩容到了 3 台机器。用户 A 连到了网关一，用户 B 连到了网关二。这个时候业务服务想给 用户 B 发消息，而这个广播的路由请求刚好打到了网关一。网关一查自己的内存 Map 发现没有 B，于是消息就凭空丢失了。
> 
> **怎么改造演进？**
> 我们后续将结合 Redis 提供解决方案。
> 当需要给用户 B 发送消息的时候不直接找本地 Map，而是直接将消息体 `Publish` 广播到一条预设的 Redis `Pub/Sub` 频道通道上。
> 我们让现存的每台网关机器只要启动，就开启一个后台协程不断从 Redis `Subscribe` 消费这条总频道。每个网关服务拉到这条消息了，就会各自在自己的本地 Map 里扫描一遍找用户，谁这里存有连接池实例，谁负责去执行 `conn.WriteMessage()` 推送给浏览器端。
> 这是分布式 WebSocket 推送架构的终极解法。
